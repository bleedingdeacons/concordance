📡 Concordance Plugin

A standalone WordPress API client for the AAGB DB Groups API. Provides transient caching, REST API proxy and WP-CLI commands.

What does Concordance do?

Concordance connects your WordPress site to the AAGBDB Groups API at https://aagbdb.org.uk/api/groups/. It fetches group data via an authenticated API key, caches responses in WordPress transients, and exposes the data through a REST proxy for JavaScript consumers and WP-CLI commands for developers and automation.

The plugin is fully standalone — it requires no external WordPress plugins. Its only Composer dependency is psr/container ^2.0 for the PSR-11 container interface.

Installation

  1. Copy the concordance/ directory into wp-content/plugins/.
  2. Run composer install from the plugin directory to install the psr/container dependency.
  3. Activate the plugin in Plugins → Installed Plugins.
  4. Go to Settings → Concordance API to verify your API key and test the connection.

On first activation, the plugin stores a default API key and cache TTL (900 seconds) in wp_options. You can change both from the settings page.

WP-CLI Commands

Concordance registers the wp concordance command when WP-CLI is active. All subcommands are injected from the container with full access to the API client and cache.

wp concordance list

List all groups from the AAGBDB API.

OptionDescription
--format=<format>Output format: table, json, csv, yaml. Default: table.
--fields=<fields>Comma-separated list of fields to display.
--limit=<n>Maximum number of groups. Default: 0 (all).
--intergroup=<id>Filter by intergroup ID.
--sort=<fields>Sort results. Comma-separated for multi-sort. Accepts: day, time, name. E.g. --sort=day,time.
--no-cacheBypass the transient cache and fetch fresh data.

wp concordance get <id>

Fetch and display a single group by ID. Supports --format and --fields.

wp concordance test

Test the API connection. Displays success with a group count, or the error message on failure.

wp concordance flush-cache

Clear all Concordance transient caches. Displays the number of deleted entries.

wp concordance config

Display the current API configuration in a table: base URL, masked API key, cache TTL, timeout, and REST namespace.

Examples

# List all groups as a table
wp concordance list

# Export as JSON, specific fields only
wp concordance list --format=json --fields=groupName,day,startTime

# First 5 groups, bypass cache
wp concordance list --limit=5 --no-cache

# Filter by intergroup ID
wp concordance list --intergroup=1

# Sort by day of week
wp concordance list --sort=day

# Sort by day then time
wp concordance list --sort=day,time

# Sort by day, time, then name
wp concordance list --sort=day,time,name --intergroup=1

# Fetch a single group
wp concordance get 42

# Test the connection
wp concordance test

# Clear all cached data
wp concordance flush-cache

# Show current configuration
wp concordance config

Boot Sequence

Concordance initialises on the plugins_loaded WordPress hook at priority 10:

WordPress loads plugin
→
Composer autoloader
→
plugins_loaded
→
Plugin::init()
→
concordance/loaded

Inside Plugin::init(), the following happens in order:

  1. A new Container (PSR-11) is created.
  2. All services are registered as lazy factories in the container.
  3. GroupListingManager is resolved — this registers the REST routes.
  4. If in admin context, SettingsAdmin is resolved — this registers the settings page.
  5. If WP-CLI is active, ConcordanceCli is resolved and registered as the wp concordance command.
  6. The concordance/loaded action fires with the container as its parameter.

PSR-11 Container

Concordance ships its own lightweight dependency injection container that implements Psr\Container\ContainerInterface. It lives at src/Core/Container.php.

How It Works

Services are registered as Closure factories. Each factory receives the container as its argument and returns the fully constructed service. Services are resolved lazily (on first get()) and cached as singletons — subsequent calls return the same instance.

// Register a service
$container->register(ApiClient::class, function (ContainerInterface $c) {
    return new ApiClient();
});

// Resolve (creates once, caches)
$client = $container->get(ApiClient::class);

// Check existence
$container->has(ApiClient::class); // true

Exception Classes

ExceptionImplementsWhen
NotFoundExceptionNotFoundExceptionInterfaceNo factory registered for the given ID
ContainerExceptionContainerExceptionInterfaceFactory throws during resolution

Global Helper

The global concordance() function returns the container singleton from anywhere after plugins_loaded:

// Resolve the API client from anywhere
$client = concordance()->get(ApiClient::class);
$groups = $client->getGroups();

// Or use the cached version
$groups = concordance()->get(ApiCache::class)->getGroups();

WordPress Hooks

concordance/loaded

Fires after all services are registered and the plugin is fully operational. Other plugins can listen on this hook to integrate with Concordance.

Parameter: Psr\Container\ContainerInterface $container

add_action('concordance/loaded', function (ContainerInterface $container) {
    $groups = $container->get(ApiCache::class)->getGroups();
    // Do something with the groups data
});
💡
Concordance boots on plugins_loaded at priority 10, so any code that runs after that (including init, wp_loaded, and template hooks) can safely call concordance().

ApiClient

The ApiClient class handles all HTTP communication with the AAGBDB API. It uses wp_remote_request internally and returns decoded JSON arrays on success or WP_Error on failure.

getGroup(int|string $groupId)array|WP_ErrorFetch a single group by ID
get(string $endpoint, array $params = [])array|WP_ErrorGeneric GET to any endpoint under /api/
post(string $endpoint, array $body = [])array|WP_ErrorGeneric POST with JSON body
testConnection()boolReturns true if the API responds successfully
getLastResponse()arrayRaw WP HTTP response from the last request (for debugging)

Direct Usage

// Via container — raw arrays
$client = concordance()->get(ApiClient::class);
$result = $client->getGroups();

if (is_wp_error($result)) {
    error_log($result->get_error_message());
} else {
    foreach ($result as $row) {
        echo $row['groupName'] . ' — ' . $row['day'] . ' ' . $row['startTime'];
    }
}

// Via container — using the GroupListing model
$result = concordance()->get(ApiCache::class)->getGroups();
$groups = GroupListing::collectionFromResponse($result);

foreach ($groups as $group) {
    echo $group->getGroupName() . ' — ' . $group->getDay() . ' ' . $group->getTimeRange();
}

// Filter by intergroup
$result = $client->getGroups(['intergroup' => 1]);

// Standalone (without container)
$client = new ApiClient('your-api-key-here');
$group  = $client->getGroup(22);

ApiCache

The ApiCache wraps ApiClient with WordPress transient caching. Cache keys are prefixed with concordance_ and hashed by the query arguments.

MethodReturnsDescription
getGroups(array $queryArgs = [], ?int $cacheTtl = null)array|WP_ErrorCached group list. TTL falls back to the stored option.
getGroup(int|string $groupId, ?int $cacheTtl = null)array|WP_ErrorCached single group lookup.
flush()intDeletes all Concordance transients. Returns count of deleted rows.
⚠️
Setting the cache TTL to 0 in the admin settings disables caching entirely — every REST request will hit the AAGBDB API directly. Use with caution on high-traffic sites.

Authentication

Concordance authenticates every API request by sending the API key in the X-Api-Key header:

X-Api-Key: SOUTHMLDS2nvQN1X...

// Headers sent with every request:
Accept: application/json
Content-Type: application/json
X-Api-Key: <your-key>

The API key is sent in the X-Api-Key header. Populated from the concordance_api_key option in wp_options. The key is never exposed to the browser — the REST proxy endpoint allows JavaScript to fetch group data without the API key leaving the server.

🚨
Keep your API key secret. Never commit it to version control or expose it in client-side code. The admin settings page renders it as a password field.

Filtering

The API returns all groups across all intergroups. Filtering by intergroup is done client-side using the intergroupId field on the GroupListing model:

// Fetch all groups, then filter by intergroup ID
$result = concordance()->get(ApiCache::class)->getGroups();
$groups = GroupListing::collectionFromResponse($result);

$filtered = array_filter(
    $groups,
    static fn(GroupListing $g) => $g->getIntergroupId() === 1
);

The WP-CLI list command supports this via the --intergroup flag: wp concordance list --intergroup=1.

💡
If the API adds new fields in the future, they are always available via getRaw() and getRawValue() without needing a model update. The WP-CLI commands use getRaw() for output so all API columns appear automatically.

REST API Proxy

Concordance exposes the AAGBDB data through two WordPress REST endpoints. These act as a proxy, keeping the API key server-side while allowing JavaScript/AJAX access.

EndpointMethodDescription
/wp-json/concordance/v1/groupsGETFetch all groups (cached). Passes query params through.
/wp-json/concordance/v1/groups/{id}GETFetch a single group by ID (cached).
💡
Both endpoints require the user to have the read capability (i.e. be logged in). To make them public, modify the permission_callback in GroupListingManager::registerRestRoutes() to return true.

Example (JavaScript)

const response = await fetch('/wp-json/concordance/v1/groups', {
    headers: { 'X-WP-Nonce': wpApiSettings.nonce }
});
const groups = await response.json();

Admin Settings

The settings page is available at Settings → Concordance API and provides three sections:

API Configuration

Two fields: the API Key (rendered as a password input) and the Cache TTL (in seconds, step 60). Both use the WordPress Settings API and are saved to wp_options.

The API key is sanitized with sanitize_text_field(); the TTL is sanitized with absint().

Connection Test

A button that triggers a live API call (nonce-protected). On success, it shows the number of groups returned. On failure, it shows the error message. This bypasses the cache — it always hits the AAGBDB API directly.

Error Handling

API Errors — WP_Error

All ApiClient methods return array|WP_Error. Non-2xx HTTP responses produce a WP_Error with code concordance_api_error, including the HTTP status code and raw response body in the error data.

JSON parse failures produce concordance_json_error. PHP exceptions produce concordance_exception. All errors are logged with error_log().

Container — NotFoundException

Thrown when Container::get() is called with a service ID that has no registered factory. The message includes the service class name.

Container — ContainerException

Thrown when a registered factory throws during resolution. Wraps the original exception with the service ID in the message.

RuntimeException — "Plugin not initialized"

Thrown by Plugin::getContainer() or concordance() when called before plugins_loaded has fired. Ensure you only access the container after WordPress has loaded plugins.

All exceptions during initialisation are caught, logged to the WordPress error log with full stack traces, and displayed as admin notices.

Troubleshooting

HTTP 401 from the API

Your API key is missing or invalid. Go to Settings → Concordance API, re-enter the key, and use the "Test API Connection" button to verify.

"Plugin not initialized" when calling concordance()

You're accessing the container too early. Move your code to run on or after plugins_loaded at priority 10 or later, or use the concordance/loaded hook.

Stale data after API changes

Data is cached in WordPress transients. Either wait for the TTL to expire, run wp concordance flush-cache, or set the cache TTL to 0 temporarily in the settings.

REST endpoints return 401

The REST proxy requires the read capability (logged-in users). Send a valid X-WP-Nonce header, or modify the permission callback in GroupListingManager to allow public access.

Class not found errors after installation

Run composer install in the plugin directory. The psr/container package must be present in vendor/ for the PSR-11 interfaces to be available.

FAQ

Does Concordance require Unity or any other plugin?

No. Concordance is fully standalone. It ships its own PSR-11 container and has no WordPress plugin dependencies. Its only Composer dependency is psr/container ^2.0.

Are services singletons?

Yes. The container calls each factory only once. After that, the same instance is returned for every get() call.

What's the concordance() helper function?

A global shorthand for Plugin::getContainer(). It returns the PSR-11 container so you can resolve services from anywhere after plugins_loaded.

Can I use the ApiClient without WordPress?

Not directly — it depends on wp_remote_request and related WordPress HTTP functions. However, the class is cleanly structured and could be adapted to use Guzzle or any other HTTP client with minimal changes.

How do I change the authentication method?

Edit ApiClient::buildHeaders(). The default sends X-Api-Key: <key>. If your API uses a different scheme (e.g. Authorization: Bearer or a query parameter), change the header construction there.

Can I add more API endpoints?

Yes. Use ApiClient::get() or ApiClient::post() with any path relative to the base URL. For example: $client->get('/meetings/'). For caching, extend ApiCache with a similar pattern to the existing getGroups() method.

How does the build script work?

The build.php script creates a ZIP archive of the plugin, excluding development files (tests, IDE configs, docs). It automatically runs composer install --no-dev if the vendor directory is missing. Use php build.php --help for full usage details.
